# ADR 0003: Define domain model persistence strategy

## Status

Accepted

## Date

2026-07-14

## Context

Spendly has an immutable Domain model containing strongly typed identifiers,
`Currency`, `Money`, persisted enums, UTC-normalized timestamps, and required
transaction relationships.

An EF Core compatibility spike proved that Npgsql can persist and materialize
this model without public setters, persistence attributes, public persistence
constructors, or navigation properties added only for EF Core.

Before adding the production `SpendlyDbContext`, migrations, and Application use
cases, the project needs one stable storage contract. Otherwise, individual
features could choose incompatible column types, enum representations, names,
delete behavior, or migration policies.

This ADR defines that contract. It does not create the production persistence
layer.

## Decision

### Ownership and boundaries

All EF Core and PostgreSQL-specific code belongs to
`Spendly.Infrastructure`.

`Spendly.Domain` remains persistence-ignorant and must not reference EF Core,
Npgsql, migrations, database attributes, or repository implementations.

The Domain owns business invariants. PostgreSQL additionally enforces cheap
structural constraints that protect data written outside normal application
paths.

### Approved mapping

| Domain value | PostgreSQL representation | EF Core mapping |
| --- | --- | --- |
| `WalletId`, `CategoryId`, `TransactionId` | `uuid` | Explicit `ValueConverter<TId, Guid>` and `ValueGeneratedNever()` |
| `Currency` | `character varying(3)` | Explicit `ValueConverter<Currency, string>` |
| `Money` | Separate `amount` and `currency_code` columns | Required complex property mapped to the containing table |
| `Money.Amount` | `numeric(19,4)` | `decimal` with `Money.Precision` and `Money.Scale` |
| persisted enums | `smallint` | Explicit enum-to-`short` converters |
| `DateTimeOffset` instants | `timestamp with time zone` | UTC values only |
| `Transaction.UpdatedAt` | nullable `timestamp with time zone` | `DateTimeOffset?` |

### Strongly typed identifiers

Identifiers are generated by the Domain with UUID version 7. PostgreSQL must
not generate or replace them.

No custom `ValueComparer` is added. The identifiers are immutable
`readonly record struct` values with value equality, so normal EF Core equality
and snapshot behavior is sufficient.

### Currency

The converter writes `Currency.Code` and materializes through `Currency.From`.
Every currency column is required unless a future domain concept explicitly
allows no currency.

Currency columns are limited to three characters and protected by a check
constraint equivalent to:

```sql
currency_code ~ '^[A-Z]{3}$'
```

`character(3)` is not used because PostgreSQL blank-pads fixed-length values.
No custom comparer is needed because `Currency` is immutable and implements
value equality.

### Money

`Money` is flattened into queryable relational columns:

```text
amount        numeric(19,4)
currency_code character varying(3)
```

It is not stored as JSON, a PostgreSQL composite type, or a serialized string.
Monetary values remain `decimal` throughout the .NET and persistence paths and
must never pass through `float` or `double`.

For transactions, PostgreSQL additionally enforces `amount > 0`.

The rule that transaction currency equals wallet currency remains a Domain and
Application responsibility. A trigger or a denormalized compound relationship
requires a separate decision and a demonstrated need.

### Precision and scale

All monetary amount columns use the current Domain policy:

```text
precision: 19
scale: 4
maximum: 999999999999999.9999
```

Persistence configuration uses `Money.Precision` and `Money.Scale` so the
Domain and database model cannot drift silently.

### Enums

`WalletType`, `CategoryType`, and `TransactionType` are stored as checked
PostgreSQL `smallint` values.

Their numeric C# values are part of the persistence contract:

- every member has an explicit numeric value;
- an existing value must never change or be reused;
- new members receive new values;
- renaming a member does not change its numeric value;
- each table has a check constraint allowing only defined values.

PostgreSQL native enum types and string enum storage are not used.

### Date and time

Domain instants are normalized to UTC before persistence and stored as
`timestamp with time zone` (`timestamptz`). The stored value represents an
instant and does not preserve the original client offset or time-zone
identifier.

Current `CreatedAt` and `OccurredAt` values are required.

`Transaction.UpdatedAt` is nullable:

```text
updated_at timestamp with time zone NULL
```

`NULL` means that the entity has not been updated since creation. If a feature
needs a civil time zone, its IANA time-zone identifier is stored separately.

### Naming conventions

All physical PostgreSQL identifiers use lowercase `snake_case` and explicit
names in EF Core configuration.

Tables use plural nouns: `wallets`, `categories`, and `transactions`.

Constraint and index patterns are:

```text
pk_<table>
fk_<dependent_table>_<principal_table>_<column_list>
ck_<table>_<rule>
ix_<table>_<column_list>
ux_<table>_<column_list>
```

Examples:

```text
pk_transactions
fk_transactions_wallets_wallet_id
ck_transactions_amount_positive
ix_transactions_wallet_id
```

Foreign-key columns are indexed. Other indexes are introduced from measured or
known query needs rather than speculatively.

### Delete behavior

Required relationships from transactions to wallets and categories use
`DeleteBehavior.Restrict`, producing PostgreSQL `ON DELETE RESTRICT`.

Deleting a wallet or category must fail while transactions reference it.
Financial history must not disappear through cascade deletion. User-facing
removal should later use an explicit archive or deactivate use case.

Cascade delete is allowed only for future dependent data whose lifecycle is
strictly owned by its principal and whose deletion cannot erase financial
history.

### Repositories and persistence abstractions

No repositories are created before real Application use cases exist.

The project will not introduce a generic CRUD repository. When a use case needs
persistence, Application defines the smallest domain-specific port and
Infrastructure implements it with EF Core. Read-side ports may return dedicated
projections instead of aggregates.

A separate unit-of-work abstraction is introduced only when an Application use
case demonstrates the need for an explicit transaction boundary.

### Migrations and deployment

The API and Worker must not call `Database.Migrate()` or
`Database.MigrateAsync()` automatically during startup.

Production migrations are applied as an explicit deployment step using a
reviewed SQL script or EF Core migration bundle. The runtime application
identity should not require schema-alteration permissions.

When production persistence is introduced:

- migrations live with Infrastructure unless a separate migrations assembly
  becomes operationally necessary;
- CI creates a clean PostgreSQL database and applies all migrations;
- migration tests use `MigrateAsync()`, not `EnsureCreated()`;
- destructive or data-rewriting migrations require a rollout and rollback plan.

### Integration testing

Database integration tests use the real Npgsql provider and PostgreSQL through
Testcontainers.

EF Core InMemory and SQLite are not substitutes for PostgreSQL persistence
verification because they do not exercise the selected PostgreSQL types,
constraints, SQL generation, or migrations.

The current compatibility spike may continue using `EnsureCreated()` until the
production context and first migration exist. After that point, production
mapping tests apply migrations and the temporary compatibility context is
removed.

The test suite protects at least:

- converters and physical column types;
- monetary precision and scale;
- enum and currency check constraints;
- UTC timestamp mapping;
- explicit key, constraint, and index names;
- restrictive foreign keys;
- PostgreSQL round-trip materialization in a new no-tracking context.

## Consequences

Positive consequences:

- all future features share one persistence contract;
- the Domain remains independent from infrastructure frameworks;
- financial values retain exact decimal representation;
- schema names and migration diffs are predictable;
- accidental cascade deletion of financial history is prevented;
- production schema changes remain controlled deployment operations;
- provider-specific behavior is verified before release.

Trade-offs:

- explicit mappings and names require more configuration code;
- numeric enum values become permanent compatibility commitments;
- selected Domain invariants are intentionally duplicated as database checks;
- original timestamp offsets are not retained;
- PostgreSQL integration tests are slower and require a container runtime.

## Alternatives considered

- **Text identifiers:** rejected because PostgreSQL has a native `uuid` type.
- **A comparer for every converter:** rejected because the current converted
  values are immutable and already have correct equality.
- **`character(3)` for Currency:** rejected because it blank-pads values.
- **JSON or strings for Money:** rejected because amount and currency need
  independent querying and constraints.
- **String or PostgreSQL-native enums:** rejected to avoid stored CLR names and
  provider-specific enum schema lifecycle.
- **Cascade delete for transactions:** rejected because it can erase financial
  history.
- **Generic repositories before use cases:** rejected as speculative CRUD
  abstraction.
- **Startup migrations:** rejected because application startup is not a
  controlled schema deployment process.
- **InMemory or SQLite database tests:** rejected because they cannot validate
  the PostgreSQL-specific contract.

## Reconsideration triggers

Revisit this ADR when a real use case requires a different persistence
abstraction, a different monetary policy, cross-table currency enforcement, a
civil time-zone identifier, a dedicated PostgreSQL schema, or new indexes based
on query plans.

A change that alters existing stored data or schema compatibility must
supersede this ADR or add a focused follow-up ADR.
